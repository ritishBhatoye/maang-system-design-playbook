# Self-Evaluation Rubric

> **面试后自我评估的评分标准。**

---

## 评分维度

| 维度 | 权重 | 描述 |
|------|------|------|
| 需求澄清 | 15% | 主动问问题、定义边界 |
| 容量估算 | 15% | 有数字、有依据 |
| 架构设计 | 25% | 组件完整、合理 |
| 深度 | 20% | 回答追问能力 |
| 故障处理 | 15% | 多层次考虑 |
| 沟通 | 10% | 清晰、检查确认 |

---

## 评分标准

### 1. 需求澄清 (15%)

| 分数 | 描述 |
|------|------|
| **5** | 主动问 5+ 个问题，覆盖规模、功能、边界 |
| **4** | 问了主要问题，理解核心需求 |
| **3** | 被动等待面试官给需求 |
| **2** | 没有问需求就动手 |
| **1** | 完全误解需求 |

### 2. 容量估算 (15%)

| 分数 | 描述 |
|------|------|
| **5** | 精确计算 QPS、存储、带宽，给出峰值 |
| **4** | 有估算，数字合理 |
| **3** | 有估算但粗糙 |
| **2** | 没有估算 |
| **1** | 数字明显不合理 |

### 3. 架构设计 (25%)

| 分数 | 描述 |
|------|------|
| **5** | 架构完整、有分层、合理、能应对扩展 |
| **4** | 架构基本完整 |
| **3** | 有明显遗漏 |
| **2** | 架构不完整 |
| **1** | 架构根本错误 |

### 4. 深度 (20%)

| 分数 | 描述 |
|------|------|
| **5** | 能回答 4+ 深度追问，举一反三 |
| **4** | 能回答 2-3 个追问 |
| **3** | 能回答 1 个追问 |
| **2** | 被问住，无法回答 |
| **1** | 明显错误 |

### 5. 故障处理 (15%)

| 分数 | 描述 |
|------|------|
| **5** | 主动讨论多种故障场景，有降级策略 |
| **4** | 能讨论基本故障情况 |
| **3** | 被追问才说 |
| **2** | 完全没有考虑 |
| **1** | 明显不合理 |

### 6. 沟通 (10%)

| 分数 | 描述 |
|------|------|
| **5** | 清晰、检查确认、互动好 |
| **4** | 基本清晰 |
| **3** | 有时混乱 |
| **2** | 沟通困难 |
| **1** | 完全无法理解 |

---

## 计算总分

```
总分 = (需求 × 0.15) + (容量 × 0.15) + (架构 × 0.25) + (深度 × 0.20) + (故障 × 0.15) + (沟通 × 0.10)
```

---

## 评级标准

| 总分 | 评级 | 建议 |
|------|------|------|
| **4.5 - 5.0** | 顶级 (L6+) | 准备充分，冲击高级别 |
| **4.0 - 4.4** | 优秀 (L5) | 可以冲击 Senior |
| **3.5 - 3.9** | 良好 (L4/L5) | 需要针对性练习 |
| **3.0 - 3.4** | 及格 | 基础不够扎实 |
| **< 3.0** | 不及格 | 需要系统学习 |

---

## 常见问题诊断

### 问题：架构设计得分低

**原因**：
- 没有使用标准模式
- 缺少关键组件
- 组件关系不清晰

**改进**：
- 练习画标准架构图
- 记住基础组件：LB、API、DB、Cache、Queue
- 了解组件之间的调用关系

### 问题：深度得分低

**原因**：
- 只知道"是什么"，不知道"为什么"
- 没有深入了解底层原理
- 缺少实际经验

**改进**：
- 每个技术问自己 3 个"为什么"
- 阅读源码或原理文章
- 了解常见问题的解决方案

### 问题：故障处理得分低

**原因**：
- 只考虑成功路径
- 没有考虑边界情况
- 缺少高可用思维

**改进**：
- 每次设计后强制思考 3 种故障
- 学习常见的故障模式
- 参考 Netflix/Google 的实践
